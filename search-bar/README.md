# Groupie tracker - Search bar

Поисковая строка Groupie Tracker выполняет поиск по нескольким типам данных, показывает серверные подсказки и исправляет умеренные опечатки через bounded Levenshtein fallback. Результаты обновляются без полной перезагрузки страницы.

## 🔎 Область поиска

Один запрос может искать по всем полям сразу или по конкретному типу выбранной подсказки:
```
| `search_type`   | Данные                      |
|-----------------|-----------------------------|
| пусто           | Все поддерживаемые поля     |
| `artist`        | Название артиста или группы |
| `member`        | Имя участника               |
| `location`      | Концертная локация          |
| `first_album`   | Дата первого альбома        |
| `creation_date` | Год создания                |
```

Свободный ввод отправляет только `q` и ищет по всем полям. При выборе подсказки вместе с `q` передаётся её точный `search_type`, поэтому одинаковый текст не смешивает разные сущности.

## 💡 Подсказки

Endpoint:

```text
GET /api/search/suggestions?q=<query>
```

Кандидаты собираются из текущего серверного каталога:

- названия артистов;
- имена участников;
- концертные локации;
- даты первых альбомов;
- годы создания.

Каждая подсказка содержит:

```json
{
  "value": "Freddie Mercury",
  "label": "Freddie Mercury - member",
  "type": "member"
}
```

Особенности:

- максимум 10 результатов;
- дубликаты удаляются по паре `type + normalized value`;
- одинаковый текст разных типов может остаться отдельными подсказками;
- пустой запрос возвращает пустой массив;
- выбор подсказки сразу запускает типизированный поиск.

## 📊 Ранжирование

Подсказки сортируются по качеству совпадения:

1. Полное совпадение.
2. Совпадение с начала строки.
3. Подстрока.
4. Совпадение отдельных токенов.
5. Fuzzy-совпадение по Левенштейну.

Внутри одинакового уровня учитываются:

- большее количество совпавших токенов;
- меньшая редакционная дистанция;
- приоритет типа `artist → member → location → first album → creation date`;
- стабильная алфавитная сортировка.

Серверный поиск использует ту же последовательность fallback-ов: phrase → token → fuzzy. Более широкий режим включается только тогда, когда предыдущий не нашёл ни одного результата во всём каталоге.

Это предотвращает ситуацию, когда неточная подсказка вытесняет существующее точное совпадение.

## 🧠 Fuzzy Levenshtein search

Fuzzy-поиск основан на ограниченном расстоянии Левенштейна: минимальном количестве вставок, удалений и замен символов.

Допустимая дистанция зависит от длины нормализованного запроса:
```
| Длина запроса | Максимальная дистанция |
|--------------:|-----------------------:|
| 1–3 символа   | 0 — fuzzy отключён     |
| 4–6 символов  | 1 правка               |
| 7–12 символов | 2 правки               |
| 13+ символов  | 3 правки               |
```

Примеры:

```text
quuen       → queen
mettalica   → metallica
freddy mercury → freddie mercury
```

### Оптимизации

- алгоритм работает по Unicode rune, а не по байтам UTF-8;
- если разница длин уже больше лимита, сравнение сразу прекращается;
- если минимальное значение текущей строки матрицы превышает лимит, дальнейший расчёт не выполняется;
- хранится только предыдущая и текущая строка матрицы;
- один токен сравнивается с отдельными токенами кандидата;
- многословный запрос сравнивается с окнами токенов и полной строкой;
- fuzzy fallback включается только для запросов, содержащих буквы;
- короткие запросы не получают опасно широкий fuzzy-результат.

## 🧹 Нормализация

Перед поиском:

- удаляются крайние пробелы;
- текст приводится к нижнему регистру;
- повторяющиеся пробелы схлопываются;
- `, . - _ / \\ – —` заменяются пробелами;
- повторяющиеся query tokens удаляются;
- односимвольные токены не участвуют в token fallback.

Поэтому варианты:

```text
AC/DC
ac-dc
ac_dc
AC   DC
```

сопоставляются с одной нормализованной формой.

Исходное отображаемое значение при этом не меняется.

## ⚡ Асинхронное поведение

Frontend ждёт `175 ms` после ввода перед запросом подсказок.

- новый ввод сбрасывает debounce timer;
- предыдущий HTTP-запрос отменяется через `AbortController`;
- каждому запросу присваивается token;
- ответ обрабатывается только если token всё ещё актуален;
- сетевые ошибки не мешают обычной отправке формы;
- при пустом вводе подсказки и `search_type` очищаются.

После submit `results.js`:

1. Формирует URL, сохраняя активные фильтры.
2. Получает серверный HTML.
3. Парсит его через `DOMParser`.
4. Заменяет только `[data-results-container]`.
5. Добавляет URL в History API.
6. Синхронизирует search и filter controls через custom event.

## ⌨️ Управление и доступность

Поиск реализует combobox pattern:

- `role="search"` у формы;
- `role="combobox"` у input;
- `role="listbox"` у списка;
- `role="option"` у каждой подсказки;
- `aria-expanded`, `aria-controls`, `aria-activedescendant` и `aria-selected` обновляются динамически;
- скрытый `role="status"` с `aria-live="polite"` сообщает число подсказок или ошибку.

Клавиатура:
```
| Клавиша     | Действие                                                 |
|-------------|----------------------------------------------------------|
| `ArrowDown` | Следующая подсказка                                      |
| `ArrowUp`   | Предыдущая подсказка                                     |
| `Enter`     | Выбрать активную подсказку или выполнить свободный поиск |
| `Escape`    | Закрыть список                                           |
| `Tab`       | Закрыть список и продолжить навигацию                    |
```
Pointer selection выполняется на `pointerdown`, чтобы выбор не терялся из-за blur input.

## 🔗 Интеграция с фильтрами и URL

Основные параметры:

```text
q=<search text>
search_type=artist|member|location|first_album|creation_date
```

Поиск не удаляет параметры фильтров. Фильтры, в свою очередь, сохраняют `q` и `search_type`.

Дополнительное поведение:

- очистка search input удаляет только `q` и `search_type`;
- пустые параметры не остаются в URL;
- Back / Forward повторно загружают только результаты;
- `pageshow` синхронизирует input после возврата из browser cache;
- прямое открытие URL даёт тот же результат через SSR;
- search form присутствует и на странице артиста, откуда запрос ведёт на главную выдачу.

## ✅ Устойчивость

- неизвестный `search_type` отклоняется как `400`;
- максимальная длина input ограничена `200` символами;
- JSON подсказок проверяется перед отображением;
- допускаются только известные типы подсказок;
- устаревшие ответы игнорируются;
- ошибки suggestions endpoint показывают скрытый статус `Suggestions unavailable`;
- отказ JavaScript не блокирует обычную GET-форму поиска;
- сервер остаётся единственным источником логики совпадения.

## 🧪 Тестирование

Тесты покрывают:

- поиск по каждому типу данных;
- свободный поиск по всем полям;
- phrase, token и fuzzy fallback;
- отсутствие fuzzy при найденном точном совпадении;
- Unicode и punctuation normalization;
- bounded Levenshtein и ранний выход;
- пороги редакционной дистанции;
- дедупликацию и лимит подсказок;
- стабильное ранжирование;
- контракты JSON endpoint;
- ARIA-разметку combobox;
- debounce, AbortController и request token contracts;
- partial update и History API.
