# Groupie tracker - Filters

Фильтры Groupie Tracker позволяют одновременно ограничивать артистов по году создания, году первого альбома, количеству участников и концертным локациям. Состояние хранится в URL, а результаты обновляются без полной перезагрузки страницы.

## 📝 Возможности

Панель `REFINE EXPLORATION` содержит:

- dual-range для года создания;
- dual-range для года первого альбома;
- кнопки количества участников;
- вариант `8+` для групп с восемью и более участниками;
- список концертных локаций с checkbox-ами;
- отдельную строку `Find city...` для быстрого поиска внутри списка.

Фильтры применяются сразу после изменения checkbox или завершения изменения range — отдельная кнопка Apply не требуется.

---

## ⚙️ Логика объединения

Категории объединяются через логическое `AND`:

```text
creation range
AND first album range
AND member count
AND location
AND search query
```

Несколько значений внутри одной категории объединяются через `OR`:

```text
members=2 OR members=4 OR members>=8
```

```text
location=London, UK OR location=Paris, France
```

Поиск и фильтры используют один серверный `Criteria`, поэтому результат не расходится между SSR, частичным обновлением и прямым открытием URL.

## 📅 Диапазоны дат

### Creation year

Два наложенных `input[type="range"]` задают минимальный и максимальный год создания.

- значения ограничиваются фактическим диапазоном каталога;
- нижний slider не может оказаться правее верхнего;
- активная часть track визуально выделяется;
- текущие значения выводятся рядом с range;
- параметры не добавляются в URL, если выбран полный доступный диапазон.

### First album year

Интерфейс показывает только годы, но сервер сравнивает полноценные даты:

- нижняя граница преобразуется в `YYYY-01-01`;
- верхняя граница преобразуется в `YYYY-12-31`;
- границы включены в результат.

Так весь выбранный год учитывается без потери альбомов внутри года.

---

## 👥 Количество участников

Интерфейс использует стабильную шкалу `1–7` и `8+`, независимо от того, какие размеры групп присутствуют в текущем ответе API.

- `1–7` означают точное количество участников;
- `8+` означает `len(Members) >= 8`;
- можно выбрать несколько значений;
- выбранные кнопки имеют отдельные visual и focus состояния;
- значение `8+` сохраняется в URL именно в таком виде и не превращается в `8`.

## 📍 Локации

Локации строятся из `/relation` и нормализуются заранее.

- удаляются дубликаты;
- значения сортируются стабильно;
- полные локации отображаются в читаемом виде `City, Region, Country`;
- служебные suffix-варианты и известные region-only значения исключаются из списка;
- выбранная локация сопоставляется со всей иерархией локации артиста.

Например, внутреннее представление может содержать:

```text
seattle, washington, usa
washington, usa
usa
```

Это позволяет серверу корректно сопоставлять нормализованные части локации.

### Поиск внутри списка

Поле `Find city...` работает только на клиенте:

- не меняет URL;
- не запускает серверную фильтрацию;
- скрывает только неподходящие checkbox-строки;
- нормализует регистр, пробелы и разделители `, . - _ / \\`;
- показывает `No matching locations`, если локальных совпадений нет.

Выбранные, но временно скрытые checkbox-ы остаются выбранными.

## 🔄 Динамические опции

Endpoint `GET /api/filter-options` возвращает:

- минимальный и максимальный год создания;
- минимальный и максимальный год первого альбома;
- фиксированную шкалу количества участников `1–7` и `8+`;
- доступные полные локации.

Frontend:

1. Запрашивает опции с `cache: no-store`.
2. Проверяет структуру JSON и допустимость значений.
3. Перестраивает range, member buttons и location list.
4. Сохраняет уже выбранные значения.
5. Повторяет загрузку после события `groupie:catalog-refreshed`.

Поэтому панель автоматически соответствует новым данным после ручного обновления кеша.

## ⚡ Частичное обновление

`results.js` отправляет обычный `GET /?...`, получает серверный HTML и заменяет только `[data-results-container]`.

- layout, top-bar и sidebar не перерисовываются;
- применяется тот же SSR-handler, поэтому отдельной JSON-модели карточек нет;
- предыдущий запрос отменяется через `AbortController`;
- request ID защищает от ответа, пришедшего не по порядку;
- URL обновляется через `history.pushState`;
- событие `popstate` поддерживает Back / Forward;
- после навигации контролы синхронизируются с URL;
- при ошибке partial update существующая страница остаётся на месте.

Если модуль частичного обновления недоступен, но `filters.js` загружен, форма выполняет обычный GET submit. Полноценный режим фильтров без JavaScript не реализован: range-параметры формируются клиентским скриптом.

## 🔗 URL-параметры
```
| Параметр        | Формат                   | Назначение                                       |
|-----------------|--------------------------|--------------------------------------------------|
| `creation_from` | `YYYY`                   | Минимальный год создания                         |
| `creation_to`   | `YYYY`                   | Максимальный год создания                        |
| `album_from`    | `YYYY-MM-DD`             | Минимальная дата первого альбома                 |
| `album_to`      | `YYYY-MM-DD`             | Максимальная дата первого альбома                |
| `members`       | повторяемое `N` или `8+` | Количество участников                            |
| `locations`     | повторяемая строка       | Концертная локация                               |
| `q`             | строка                   | Поисковый запрос, сохраняется вместе с фильтрами |
| `search_type`   | enum                     | Тип выбранной поисковой подсказки                |
```

Пример:

```text
/?q=queen&creation_from=1970&members=4&locations=London%2C+UK
```

URL можно сохранить, открыть в новой вкладке или отправить другому пользователю — сервер восстановит тот же результат без зависимости от клиентского состояния.

## ✅ Валидация и устойчивость

Backend проверяет:

- корректность числовых годов;
- порядок `from <= to`;
- формат album date;
- положительное количество участников;
- допустимость `8+`;
- известные значения `search_type`;
- нормализацию выбранных локаций и сведение повторяющихся условий к наборам при фильтрации.

Некорректный запрос получает фирменную страницу `400`, а не частично применённые условия.

Frontend дополнительно:

- не отправляет пустые параметры;
- не отправляет границы, равные полному диапазону;
- различает доверенное пользовательское изменение и программную синхронизацию;
- восстанавливает checkbox и range после Back / Forward и `pageshow`.

## 🧪 Тестирование

Тесты покрывают:

- отдельные и совместные диапазоны;
- включённые граничные значения;
- точное количество участников и `8+`;
- несколько member/location параметров;
- нормализацию локаций и иерархическое совпадение;
- дедупликацию и сортировку динамических опций;
- исключение region-only и suffix-локаций;
- JSON-контракт `/api/filter-options`;
- HTML-разметку фильтров;
- JavaScript-контракты partial update, URL state и refresh event.
